\section{Parallel Planning Approach}
\label{sec:parallel_planning_approach}
\hspace{\parindent}
The proposed architecture extends the original TEA*-based planner by executing several
independent TEA* instances in parallel. As illustrated in
Figure~\ref{fig:parallel_planning_framework}, the Supervisor sends a planning request to the
Path Planner containing the current robot positions, their corresponding goals, and the maximum
time available for planning, denoted by
\(I=\{(p_i,g_i)\}_{i=1}^{N}, T_{\max}\).

After receiving the request, the Path Planner initializes the execution and spawns a set of worker
threads. Each thread retrieves a heuristic from a shared heuristic pool, uses it to compute a
priority permutation, and then runs TEA* independently using that ordering. Each execution
therefore produces a candidate plan associated with one priority ordering.

\begin{figure}[H]
    \centering
    \includegraphics[width=1.0\linewidth]{parallel_planning_framework.png}
    \caption{Overview of the proposed parallel planning framework.}
    \label{fig:parallel_planning_framework}
\end{figure}


For simplicity, Figure~\ref{fig:parallel_planning_framework} represents the case in which the
number of available heuristics is equal to the number of worker threads. In that simplified case,
each thread retrieves one heuristic and produces one candidate plan. In the general case, however, these numbers do not need to be equal, since a thread may evaluate several heuristics during the same planning cycle as long as planning time is still available and heuristics remain to be tested.

During the parallel execution stage, each worker thread generates one or more candidate plans, which are collected in a set \(P\) and subsequently evaluated using the selected cost function \(J\). Once all candidate plans have been evaluated, the plan with the lowest cost is returned to the Supervisor. In this work, \(J(P_i)\) is defined as the sum of the individual costs of all robot paths in candidate plan \(P_i\). Candidate plans that do not include valid paths for all robots are assigned an \(\infty\) cost, since a plan is considered to have failed whenever at least one robot is unable to obtain a valid path.

The following subsections describe these components in more detail, namely the thread implementation details, the heuristic dispatching strategy, the termination mechanisms, and the handling of redundant heuristics.

\subsection{Thread Workers Implementation Details}
\label{subsec:thread_implementation_details}
\hspace{\parindent}
The parallel framework was implemented by exploiting the existing Object-Oriented Programming
(OOP) design of the TEA* planner. In the original implementation, TEA* was already encapsulated
as a class, with its own internal planning state and methods. Therefore, the adopted solution was
to implement the TEA* Parallel Planner class, illustrated in
Figure~\ref{fig:parallel_planner_class_diagram}, which manages vectors of independent TEA*
and A* instances. The TEA* instances are used by the worker threads to evaluate different
priority orderings, while the A* instances support heuristics that require single-robot shortest-path
computations. This structure allows each worker to operate on separate planner objects while
preserving the original sequential behaviour of TEA* within each individual execution.

\begin{figure}[H]
    \centering
    \includegraphics[width=1.0\linewidth]{class_diagram.png}
    \caption{UML Class Diagram representing the implemented parallel planning framework and its main components.}
    \label{fig:parallel_planner_class_diagram}
\end{figure}

Different thread-based parallelization options could be considered for this type of implementation,
including \textit{OpenMP}, \textit{Intel TBB}, and the C++ standard threading library.
These approaches rely on kernel-level threads that can be scheduled on multiple processor cores,
but they expose different programming models. \textit{Intel TBB} provides flexible task-based
parallelism, while the C++ standard threading library offers explicit thread management through
\textit{std::thread}. Although these alternatives provide a high degree of control, the proposed
framework does not require such fine-grained control, since the main parallel workload is already
structured as a set of independent worker executions.
For this reason, \textit{OpenMP} was used to launch the main worker threads. It provides a
higher-level and sufficiently expressive interface for distributing these workers across multiple
threads, while keeping the implementation simple and avoiding explicit management of thread
creation, joining, or scheduling.

During each planning cycle, the worker executions are independent but not equivalent. Since each
worker evaluates a different priority ordering, each TEA* instance may produce a different sequence
of reservations, node expansions, and path rejections. This means that, although the workers follow
the same planning algorithm, their internal states evolve differently throughout the parallel execution.
To avoid interference between concurrent executions, the mutable internal state of TEA* is kept
private to each planner instance. This includes structures such as the 3D costmap, where the
space-time reservations of the planned robot paths are stored, as well as the lists used during the
search. Although it would be possible to design a shared version of these structures, doing so would
require frequent synchronization during the most intensive parts of the algorithm. Since these
accesses occur repeatedly throughout TEA*, protecting them with mutexes would increase
implementation complexity and could significantly reduce the benefit of parallel execution.
Therefore, only data that remains constant during planning is shared between instances. For
example, the precomputed map cost lookup table is shared, since it is read-only and can be safely
accessed by multiple threads without synchronization.

Overall, the implementation combines the simplicity of \textit{OpenMP} for launching the main worker
threads with explicit synchronization mechanisms only for the few shared structures that require
controlled access. This is the case of the heuristic dispatching mechanism, which is responsible
for assigning priority orderings to the worker threads and is described in the next subsection.
This design keeps the core TEA* algorithm unchanged, while enabling several priority orderings
to be evaluated concurrently through independent planner instances.


\hspace{\parindent}\subsection{Heuristic Dispatching Mechanism}
\label{subsec:heuristic_dispatching}
\hspace{\parindent}
The dispatching mechanism was designed to avoid assigning highly redundant heuristics to the
worker threads at the same time. Since the heuristic set is organized into different families, the
objective is to promote a balanced use of these families during the parallel execution. This increases
the diversity of the generated priority orderings and reduces the probability that several threads
evaluate very similar planning alternatives. The overall dispatching flow executed by each worker
thread is illustrated in Figure~\ref{fig:heuristic_dispatching_flow}.

To achieve this, the available heuristics are stored in a shared priority queue. Each heuristic is associated with a heuristic group, and each group has a counter indicating how many heuristics from that group have already been selected during the current planning cycle. The priority queue is ordered according to these group counters, so that heuristics belonging to the least-used groups are given higher priority.

Whenever a worker thread needs a new heuristic, it accesses the shared queue inside a critical section protected by a mutex and pops a heuristic from the group with the lowest usage counter. Since the selected heuristic is removed from the queue during this operation, two threads cannot evaluate the exact same heuristic during the same planning cycle. After the heuristic is selected, the counter of the corresponding group is incremented, and the queue is reordered accordingly.

This mechanism encourages the simultaneous evaluation of heuristics from different families considering that, instead of assigning several heuristics from the same family to different threads at the same time,
the dispatcher tends to distribute the available computational resources across all the strategies.

As depicted in Figure~\ref{fig:heuristic_dispatching_flow}, if the shared heuristic queue becomes empty, the worker can no longer retrieve a heuristic from the pool. In this case, if that worker has not yet obtained a valid candidate plan, an additional attempt is performed using the \textit{Exploration Conflicts Pressure} replanning heuristic. The statistics required by this heuristic are accumulated locally by each worker across its previous planning attempts, namely the number of reservation-based rejections and the number of expanded nodes already mentioned in Subsection~\ref{subsec:exploration_conflicts_pressure}. This heuristic is only used after all regular heuristic attempts of that worker have failed, since it tends to reverse the priority order in an informed way .


\begin{figure}[H]
    \centering
    \includegraphics[width=1.0\linewidth]{fluxogram.drawio.png}
    \caption{Heuristic dispatching flow executed by each worker thread.}
    \label{fig:heuristic_dispatching_flow}
\end{figure}

Although a predefined static ordering with manually alternated heuristic families could provide a
similar behaviour in the current implementation, the main advantage of the implemented mechanism
is the flexibility it provides for extending the scheduling policy. In future extensions, the dispatcher
could prioritize heuristics according to the remaining planning time or even the quality of candidate
plans obtained by previous orderings.



As shown in Figure~\ref{fig:heuristic_dispatching_flow}, after leaving the critical section, the
worker thread evaluates the selected heuristic by generating the corresponding priority ordering
and computing the associated TEA* paths. If the regular heuristic queue is empty, the worker either
returns the best available plan, when it has already found a valid one, or performs an additional
attempt with the \textit{Exploration Conflicts Pressure} heuristic. The worker only repeats this
procedure after checking whether the current planning cycle is still active, according to the timeout
and termination logic described in the following subsection.


\subsection{Timeout Handling and Controlled Termination}
\label{subsec:timeout_controlled_termination}
\hspace{\parindent}
The parallel planner was designed to operate under a maximum planning time \(T_{\max}\), which represents the time available to compute and return a valid plan after a planning request is received. In the intended system behaviour, this value is provided by the Supervisor and may be computed dynamically in future versions according to the current execution state of the robots. For instance, the Supervisor could estimate \(T_{\max}\) from the remaining time before the most urgent robot reaches its next relevant node. On the Path Planner side, the timeout mechanism is already supported as in the current implementation, \(T_{\max}\) is provided as a static parameter.

Immediately before launching the worker threads, a global timer is initialized. To monitor the elapsed
planning time, an additional lightweight monitoring thread is launched alongside the previously
described worker threads. During the planning cycle, this thread periodically compares the elapsed
time with \(T_{\max}\), sleeping between consecutive checks in order to reduce computational overhead
and avoid unnecessary CPU usage. Once the elapsed time exceeds the maximum planning time, the
monitoring thread sets a shared atomic boolean flag to indicate that the timeout has been reached.

However, the timeout does not immediately force all workers to stop. Since returning no plan is
undesirable when no valid solution has yet been found, the termination condition also depends on
the existence of at least one complete candidate plan. For this reason, another shared atomic flag is
used to indicate whether a valid plan has already been obtained. The first worker thread that finds
a complete plan, containing valid paths for all robots, sets this flag. 
The planning cycle is therefore considered inactive only when both of the following conditions are satisfied:

\begin{itemize}
    \item The maximum planning time \(T_{\max}\) has been reached;
    \item At least one valid candidate plan is available.
\end{itemize}

To allow running TEA* instances to stop in a controlled manner, the termination condition is
checked at the beginning of each TEA* step. This point was chosen because the step operation is the most frequently executed operation throughout
the search process, making it a natural location to test whether the current planning attempt should
continue. When the termination condition becomes true, each
worker detects it at the beginning of a subsequent step and interrupts its current planning process
safely, allowing local variables and internal planner state to be closed consistently before the thread
returns.

This mechanism allows the planner to use the available time to continue searching for better
candidate plans after the first valid solution is found, while still ensuring that the computation can
be interrupted once the time budget has expired. As a result, the framework does not stop at the
first feasible plan, but instead uses the remaining time to improve the final selection whenever
possible.

\subsection{Handling Redundant Heuristics}
\label{subsec:handling_redundant_heuristics}
\hspace{\parindent}
The results presented in the previous section showed that \textit{Radius Neighbour Density} and
\textit{K-hop Neighbor Density} often produce equivalent or very similar priority orderings. Since
running both heuristics may therefore lead to redundant TEA* executions, the selection between
them was based on one of their main differentiating factors, the estimated computation time. This
criterion is directly motivated by their different computational behaviours, described in
Equations~\ref{eq:radius_neighbour_density_complexity} and
\ref{eq:k_hop_neighbour_density_complexity}.

To estimate this computation time, linear regression models were fitted for both heuristics. Since
the theoretical complexity analysis already suggested the main runtime trends expected for each
case, more complex modelling approaches were not required. 


\subsubsection{Runtime Estimation for Radius Neighbour Density}
\label{subsubsec:radius_runtime_estimation}
\hspace{\parindent}
To estimate the computation time of the \textit{Radius Neighbour Density} heuristic, an experimental
dataset was collected by measuring its runtime for different numbers of robots. The tested values
were \(N=\{1,5,10,20,50,100,150,200,300,400\}\), and, for each value of \(N\), the heuristic was
executed 50 times. The measured execution times were then used to fit different regression models.

The choice of candidate models was guided by Equation~\ref{eq:radius_neighbour_density_complexity}.
Accordingly, different regression models were tested using combinations of \(N\), \(N^2\), and
\(N\log N\). Table~\ref{tab:radius_runtime_models} summarizes the obtained results.

the difference between the \(N+N^2\) and \(N^2+N\log N\) models was almost negligible. This is
consistent with the fact that the dominant runtime behaviour is captured by the \(N^2\) term, while
the remaining lower-order contribution can be sufficiently approximated by the linear term over
the tested range of robot numbers.

Figure~\ref{fig:lr_radius} shows the fitted curve obtained with the selected model. The regression

\subsubsection{Runtime Estimation for K-hop Neighbor Density}
\label{subsubsec:khop_runtime_estimation}
\hspace{\parindent}
For the \textit{K-hop Neighbor Density} heuristic, the runtime estimation had to account for the
expansion process performed when computing the topological neighbourhood of each robot. In this
case, the computation time is mainly affected by the total number of node expansions performed
during the \(k\)-hop searches.

To obtain measurements with different expansion counts, the average roadmap degree was kept
fixed, since creating new maps or modifying existing ones solely to vary this parameter would add
unnecessary experimental complexity. Instead, the total number of expansions was varied by
changing the neighbourhood depth \(k\) and the number of robots \(N\). The tested values were
\(k=\{1,2,3\}\) and \(N=\{10,25,50,100,150,200\}\). For each combination of \(N\) and \(k\), the
heuristic was executed 20 times, and the corresponding computation times were recorded.

The tested regression models considered the measured total number of node expansions, denoted
by \(E\), together with terms depending on the number of robots. Table~\ref{tab:khop_runtime_models}
summarizes the obtained results.

Among the tested alternatives, the best result was obtained by combining \(N\log N\) with the
total number of node expansions \(E\), leading to the following runtime model:

\begin{equation}
    T_{\mathrm{khop}}(N,E)
    =
    \theta_0 + \theta_1N\log N + \theta_2E .
    \label{eq:khop_runtime_model}
\end{equation}

Compared with the model using only \(E\), the inclusion of a term depending on
\(N\) improves the fit, indicating that the number of robots also contributes to the observed runtime
beyond the raw expansion count.

Figure~\ref{fig:lr_khop} shows the fitted regression plane obtained with the selected model. The
three-dimensional representation is used because the estimated computation time depends on two
input variables, \(N\log N\) and \(E\).

\begin{figure}[H]
    \centering
    \includegraphics[width=1.0\linewidth]{lr_khop.png}
    \caption{Regression model fitted to the computation time of the \textit{K-hop Neighbor Density} heuristic.}
    \label{fig:lr_khop}
\end{figure}

To use this model before executing the heuristic, the total number of node expansions \(E\) must be
estimated from the parameters of the current planning instance. In practice, the neighbourhood
depth \(k\) is fixed for a given run, and high values of \(k\) are not typically useful in this context,
since they rapidly expand the considered neighbourhood and may reduce the locality of the density
estimate.

For this reason, the expansion effort was approximated from the number of vertices reachable
within \(k_{\text{hops}}\) hops, previously defined in
Equation~\ref{eq:khop_reachable_vertices}. Assuming an average roadmap degree of \(DoF=4\),
this provides a practical estimate of the number of vertices explored around each robot. Although the recursive implementation may expand some vertices more than once, this becomes
unlikely for the small values of \(k_{\text{hops}}\) considered here, making the approximation
sufficient for runtime prediction.

Since the \(k\)-hop expansion is performed for each robot, the total expansion estimate used by the
runtime model is given by

\begin{equation}
    \hat{E}
    =
    N \cdot N_k
    =
    N \cdot \left(1 + 2k_{\text{hops}}(k_{\text{hops}}+1)\right),
    \quad DoF=4.
    \label{eq:khop_estimated_expansions}
\end{equation}

This estimated value \(\hat{E}\) is then provided to
Equation~\ref{eq:khop_runtime_model}, allowing the computation time of the
\textit{K-hop Neighbor Density} heuristic to be predicted before executing it.


\subsubsection{Runtime-Based Comparison}
\label{subsubsec:runtime_based_comparison}
\hspace{\parindent}
After fitting the runtime estimation models for both heuristics, a direct comparison was performed
to determine which one should be selected for a given planning instance. While the
\textit{Radius Neighbour Density} model depends only on the number of robots \(N\), the
\textit{K-hop Neighbor Density} model also depends on the estimated number of node expansions
\(\hat{E}\). However, from Equation~\ref{eq:khop_estimated_expansions}, it can be observed that,
once \(DoF\) and \(k_{\text{hops}}\) are fixed, \(\hat{E}\) also becomes a function of \(N\). This matches
the practical use of the model, where \(k_{\text{hops}}\) is fixed for a given run and \(\hat{E}\) is
predicted before executing the heuristic. Therefore, both runtime models can be evaluated as
functions of \(N\), allowing a direct comparison between the two estimated computation times.

Figures~\ref{fig:radius_khop_k1}, \ref{fig:radius_khop_k2},
\ref{fig:radius_khop_k3}, and \ref{fig:radius_khop_k4} show the estimated runtime comparison
for different values of \(k_{\text{hops}}\), assuming \(DoF=4\). The intersection between the two
curves indicates the approximate robot count at which both heuristics are expected to have the
same computation time.

The intersections occur approximately at \(N=54.47\), \(N=99.94\), \(N=191.62\), and
\(N=326.65\) for \(k_{\text{hops}}=1,2,3,\) and \(4\), respectively. This increasing threshold is
consistent with the fact that larger \(k_{\text{hops}}\) values require more neighbourhood
expansions, delaying the point at which the \(k\)-hop-based heuristic becomes more efficient than
the radius-based alternative.

growth of the radius-based model.
